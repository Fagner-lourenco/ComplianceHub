This method allows you to insert a file with a batch (batch) of records to be queried in the Data Platform processing queue. In return, the user receives a JobId, the unique identifier of this newly defined batch processing.

Operation

This method receives as input the parameters required to query a batch of records: the file with the records to be processed, the API to be queried, the format of the output file, and so on. These elements (described in more detail in the section below) are used to insert the job into the processing queue and, subsequently, to execute the queries.

The input file, which contains the records to be processed, can be submitted in multiple different formats (described in more detail in the section below).

For files exchanged via HTTP or HTTPS, the input file must be available via a public link that allows the file to be downloaded by the executing process; otherwise, the batch execution will fail.

For files made available through S3 Buckets, the access credentials provided in the API call must have permission to get and put objects in the bucket, in addition to identifying the bucket's region. In other words, for this access, we need permission for the methods: GetBucketLocation, GetObject, and PutObject .

It is possible to use all the query mechanisms available in the Data Platform's conventional API, whether for querying people, companies or validations, combined with the elements of the input files through the definition of a query template . This includes executing queries using alternative keys, sending reference dates for backtest execution and much more.

In addition to the mechanisms available in the API, Batch processing also supports query retry . The purpose of this mechanism is to support processing of very large files or files that depend on unstable information sources, allowing queries with errors to be re-executed without the need to resubmit the input file, simplifying the work of customers. It is important to remember that you are never charged for queries with errors, so retrying does not imply additional costs for customers. Queries are retried in two situations:

There was some connection failure or response timeout in the Query API call; or ;
All datasets queried returned an error code less than or equal to -1200 (see the error code table for details).

The optional parameters used for query retry are described in the table below.

The output file is made available through a link in the BigData Corp's file transfer area, which is located at https://files.bigdatacorp.com.br/go, and is available for download for up to 30 days after execution. of the job. Alternatively, if using the S3 Buckets file exchange method, the output file is deposited in the repository defined by the customer at the input.

ATTENTION: After calling this method, the file will be validated and loaded for processing, but the query process will only start when the StartJobExecution method, described below, is called.

Formatting the URLs of process input and output files

The syntax below applies to fields InputFileURL and OutputFileUrl, and is used to specify the file transfer mechanism, in addition to the parameters necessary for the transfer to be carried out successfully. We list the supported transfer mechanisms below, and the syntax for each one:

Transfer via HTTP or HTTPS: In this type of transfer, the input file will be downloaded from the link provided, and the output file will be generated on a BigDataCorp clipboard for download by the client. The following example illustrates how to fill in the required parameters:

{
  "InputFileUrl": "web{https://www.example.com/mysamplefile.txt}",
  "InputKeysHeader" : "Documento",
  "APIName": "people",
  "Datasets": "basic_data",
  "QueryTemplate": "doc{{0}}"
}


Transfer via S3 (AWS Data Storage Service): In this type of transfer, the customer can specify a return address independent of the incoming address. If the output file's URL parameter is not populated, the results file will be placed in the same bucket as the input file. The following example illustrates how to fill in the required parameters:

{
  "InputFileUrl": "file{s3://aws_access_key:aws_secret_key@bucket_name}, remotefilepath{path_to_file_in_s3}",
  "OutputFileUrl": "file{s3://aws_access_key:aws_secret_key@bucket_name}, remotefilepath{path_for_file_to_be_saved_at}",
  "InputKeysHeader" : "Documento",
  "InputFileDelimiter": "	",
  "InputFileHasHeader": false,
  "OutputFileDelimiter": "	",
  "APIName": "people",
  "Datasets": "basic_data",
  "QueryTemplate": "doc{{0}}"
}


FTP transfer: In this transfer model, the input and output address can be completely different. Parameters in parentheses "()" are optional, and if not filled in, they will assume default values. The following example illustrates how to fill in the required parameters:

{
  "InputFileUrl": "file{ftp://user(:password)@dns(:port)}, remotefilepath{path_to_file_in_ftp}",
  "OutputFileUrl": "file{ftp://user(:password)@dns(:port)}, remotefilepath{path_for_file_to_be_saved_at}",
  "InputKeysHeader" : "Documento",
  "InputFileDelimiter": "	",
  "InputFileHasHeader": false,
  "OutputFileDelimiter": "	",
  "APIName": "people",
  "Datasets": "basic_data",
  "QueryTemplate": "doc{{0}}"
}


Transfer via SFTP: In this transfer model, the input and output address can be completely different. Parameters in parentheses "()" are optional, and if not filled in, they will assume default values. The following example illustrates how to fill in the required parameters:

{
  "InputFileUrl": "file{sftp://user(:password)@dns(:port)}, remotefilepath{path_to_file_in_sftp}",
  "OutputFileUrl": "file{sftp://user(:password)@dns(:port)}, remotefilepath{path_for_file_to_be_saved_at}",
  "InputKeysHeader" : "Documento",
  "InputFileDelimiter": "	",
  "InputFileHasHeader": false,
  "OutputFileDelimiter": "	",
  "APIName": "people",
  "Datasets": "basic_data",
  "QueryTemplate": "doc{{0}}"
}


IMPORTANT! The fields of InputFileUrl and OutputFileUrl are independent. An input file present on S3 and an output file posted to an SFTP folder can be specified, or any other combination of the mechanisms listed above.

A password will be added to a return file depending on the type of destination environment. For internal environments such as S3, FTP, or SFTP, where the file is stored within the corporate environment, a password is not added to the file. However, for external environments, publicly accessible via URL, an automatically generated password is always included to ensure the security of the file.

If preferred, it is possible to specify the password to be added to a file via the 'OutputFilePassword' field when making a request, thus allowing files sent to external environments to have a manually defined password. It is worth noting that, if the 'OutputFilePassword' field is included in the request, but is empty, filled only with spaces or null, the password assigned to the file will be generated automatically.

In all cases where a password is applied to the file, it will be sent by email to the user's registered address and to the emails listed in the CopyEmails field, also present in the current request.

Input files
Formats .txt,.csv and .tsv.
Each line of the file must contain one record.
If the record has multiple columns, they must be separated by a delimiter of your choice (e.g. comma, semicolon, tab, etc.). Make sure the same delimiter is specified in the field InputKeysHeader and in the field InputFileDelimiter..
Format .xlsx (Excel spreadsheet).
When specifying the file path in the field InputFileUrl, if necessary, include the index of the desired spreadsheet right after the file extension, in the format [x], where x represents the worksheet index, starting at 1.
If no index is provided, the default value will be used. 1.
Example:
{"InputFileUrl":"web{https://www.example.com/mysamplefile.xlsx[2]}"}
Google Docs files.
Make sure the document is shared as a public file or has appropriate permissions for access via link.
Insert the direct link of the file in the field InputFileUrl. The system will download and process the content automatically.
Example:
{"InputFileUrl":"web{https://docs.google.com/document/d/example-file/edit?usp=sharing}"}
Return

The method returns an error code and message, in case a problem has occurred. Otherwise, the job's unique identifier is returned, which can be used to retrieve other information about the job or to cancel its execution (see the other Batch Processing API methods).

The following table describes the fields of the return object of this method.

Field	Description
JobId	Identifier of the created job
StatusCode	Call result code
StatusMessage	Descriptive call result message
TotalExecutionTime	Total execution time of the request on the server
RequestTimestamp	Date and time of completion of the request execution on the server
Status Codes

The following table describes possible return codes and messages.

Code	Message	Description
0	OK	Successful execution
-1	INVALID AUTHENTICATION PARAMETERS	The access token provided in the entry is invalid
-2	BAD REQUEST FORMAT	The sent request is not well formatted
-101	INVALID INPUT FILE URL	The link to the specified input file is not working
-102	INVALID OUTPUT FILE FORMAT	The specified output format is outside the allowed values
-103	INVALID SAMPLE RATE	The defined sample rate is outside the allowed values
-105	INVALID API NAME	The name of the API that will be queried by the process is outside the allowed values
-106	INVALID DATASETS	The selected dataset set is invalid
-107	INVALID QUERY TEMPLATE	The defined query element template is invalid
-108	INVALID JOB ID	The job identifier passed in the input is invalid
-109	INVALID JOB STATUS FOR OPERATION	The selected job cannot be paused or canceled due to its current status
-1200	UNEXPECTED ERROR. PLEASE CONTACT OUR SUPPORT TEAM	An unexpected error has occurred. Get in touch with our service team
